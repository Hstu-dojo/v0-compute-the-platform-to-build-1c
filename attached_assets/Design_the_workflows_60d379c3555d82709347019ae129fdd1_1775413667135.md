## Group 1: Authentication Flows

### 1.1 — New user registration

```mermaid
sequenceDiagram
    participant Client
    participant FirebaseAuth as Firebase Auth
    participant Backend
    participant Postgres
    participant Ledger as Token Ledger

    Client->>FirebaseAuth: Register with email/password or Google
    FirebaseAuth-->>Client: Returns firebase_uid + JWT

    Client->>Backend: POST /auth/register {jwt, display_name}
    Backend->>Backend: Verify JWT signature against Firebase public keys
    Backend->>Postgres: SELECT * FROM users WHERE firebase_uid = ?
    Postgres-->>Backend: No row found

    Backend->>Postgres: INSERT INTO users {firebase_uid, email, role: student}
    Backend->>Postgres: INSERT INTO user_profiles {user_id, defaults}
    Backend->>Postgres: INSERT INTO auth_providers {user_id, provider, provider_uid}

    Backend->>Postgres: INSERT INTO subscriptions {user_id, plan: free, status: active}
    Backend->>Ledger: INSERT INTO token_ledger {event_type: credit_grant, amount: free_plan_credits}
    Backend->>Postgres: INSERT INTO monthly_credit_grants {user_id, credits_granted, billing_period}

    Backend->>Postgres: INSERT INTO audit_logs {action: user_registered}
    Backend-->>Client: 201 Created {user_id, token_balance}
```

### 1.3 — JWT middleware validation on every request

```mermaid
flowchart TD
    A[Incoming request] --> B[Extract Authorization header]
    B --> C{Header present?}
    C -- No --> D[Return 401 Unauthorized]
    C -- Yes --> E[Verify JWT with Firebase public keys]
    E --> F{Signature valid?}
    F -- No --> G[Return 401 Invalid token]
    F -- Yes --> H{Token expired?}
    H -- Yes --> I[Return 401 Token expired]
    H -- No --> J[Extract firebase_uid from claims]
    J --> K{User in Redis cache?}
    K -- Yes --> L[Attach cached user to req.user]
    K -- No --> M[Query Postgres for user + role + plan + balance]
    M --> N{User found and active?}
    N -- No --> O[Return 403 Account not found or suspended]
    N -- Yes --> P[Write to Redis cache TTL 300s]
    P --> L
    L --> Q[Pass to next middleware]
```

```mermaid
sequenceDiagram

    participant Backend
    participant VirusScan as Virus Scanner
    participant R2 as Cloudflare R2
    participant Postgres
    participant BullMQ

    Backend->>Postgres: INSERT INTO documents {user_id, r2_path, extracted_text, content_hash, status: pending_embedding}
    Backend->>BullMQ: Enqueue job on studley:embedding queue {document_id}
```

### 2.3 — Embedding & chunking worker

```mermaid
sequenceDiagram
    participant BullMQ
    participant EmbedWorker as Embedding Worker
    participant Postgres
    participant ModelRouter as Model Router
    participant OpenAI as Embedding Provider

    end

    EmbedWorker->>Postgres: UPDATE embedding_jobs SET status: completed, tokens_used
    EmbedWorker->>Postgres: UPDATE documents SET processing_status: ready
    EmbedWorker->>Postgres: INSERT INTO token_ledger {event_type: debit_confirmed, task_type: embedding}
```

## Group 3: Study Set Generation Flows

### 3.1 — Batch creation and job fan-out

```mermaid
sequenceDiagram
    participant Client
    participant Backend
    participant QuotaGuard as Token Quota Guard
    participant Postgres
    participant BullMQ

    Client->>Backend: POST /study-sets/generate {document_id, types: [notes, flashcards, podcast]}
    Backend->>Postgres: SELECT document WHERE id = ? AND user_id = ? AND status = ready

    alt Document not ready
        Backend-->>Client: 409 Document still processing
    end

    Backend->>Postgres: SELECT model_config for each task_type WHERE is_active = true
    Backend->>Backend: Estimate credit cost per job from model_config.token_multiplier
    Backend->>Backend: Sum total estimated credits across all selected types

    Backend->>QuotaGuard: Check balance >= total_estimated_credits
    QuotaGuard->>Postgres: SELECT SUM(amount) FROM token_ledger WHERE user_id = ?

    alt Insufficient balance
        Backend-->>Client: 402 Insufficient credits
    end

    Backend->>Postgres: INSERT INTO study_set_batches {user_id, document_id, selected_types, status: processing}

    loop For each selected type
        Backend->>Postgres: INSERT INTO token_ledger {event_type: debit_pending, amount: -estimated, job_id}
        Backend->>Postgres: INSERT INTO bullmq_jobs {batch_id, queue_name, task_type, status: queued}
        Backend->>BullMQ: Enqueue job on studley:{type} {document_id, batch_id, job_id, user_id}
    end

    Backend-->>Client: 202 Accepted {batch_id, job_ids[], estimated_completion_s}
```

### 3.2 — Generic study set worker flow (notes, MCQ, flashcards, fill-in-blanks, written test, content)

```mermaid
sequenceDiagram
    participant BullMQ
    participant Worker
    participant Postgres
    participant RAGEngine as RAG Engine
    participant ModelRouter as Model Router
    participant Ledger as Token Ledger
    participant WSS as WebSocket Server

    BullMQ->>Worker: Dequeue job {document_id, batch_id, job_id, task_type}
    Worker->>Postgres: UPDATE bullmq_jobs SET status: processing, started_at: NOW()
    Worker->>WSS: Emit progress {batch_id, job_id, status: processing}

    Worker->>Postgres: SELECT output_cache WHERE content_hash = ? AND task_type = ? AND model_name = ?

    alt Cache hit
        Worker->>Postgres: UPDATE output_cache SET hit_count += 1
        Worker->>Postgres: INSERT INTO study_set_outputs {status: completed, output_data: cached}
        Worker->>Postgres: INSERT INTO token_ledger {event_type: debit_released, note: cache_hit}
        Worker->>Postgres: UPDATE bullmq_jobs SET status: completed
        Worker->>WSS: Emit progress {batch_id, job_id, status: completed}
        Worker-->>BullMQ: Job done
    end

    Worker->>RAGEngine: Vector search {document_id, task_type_query, top_k: 8}
    RAGEngine->>Postgres: SELECT chunk_text ORDER BY embedding <=> query_vector LIMIT 8
    Postgres-->>RAGEngine: Top-K chunks
    RAGEngine->>Postgres: INSERT INTO rag_contexts {job_id, chunk_ids_used, assembled_context}
    RAGEngine-->>Worker: assembled_context

    Worker->>Postgres: SELECT prompt_templates WHERE task_type = ? AND is_active = true
    Worker->>Worker: Interpolate template with assembled_context and document metadata

    Worker->>ModelRouter: Generate {task_type, prompt, user_id, job_id}
    ModelRouter-->>Worker: Returns output_data + raw_tokens + model_name + provider

    Worker->>Postgres: INSERT INTO study_set_outputs {batch_id, type, status: completed, output_data}
    Worker->>Postgres: INSERT INTO ai_request_logs {job_id, provider, model, prompt_tokens, completion_tokens, latency_ms}
    Worker->>Postgres: INSERT INTO output_cache {content_hash, task_type, model_name, output_data}

    Worker->>Ledger: INSERT debit_confirmed {job_id, raw_tokens, multiplier, computed_credits}
    Worker->>Ledger: INSERT refund if actual_credits < reserved_credits

    Worker->>Postgres: UPDATE bullmq_jobs SET status: completed, completed_at: NOW()
    Worker->>Postgres: UPDATE study_set_batches SET completed_jobs += 1
    Worker->>Postgres: UPDATE usage_stats {total_jobs_run, total_credits_used}

    alt All jobs in batch completed
        Worker->>Postgres: UPDATE study_set_batches SET status: completed
    end

    Worker->>WSS: Emit progress {batch_id, job_id, status: completed, output_id}
```

### 3.4 — Job retry and dead letter flow

```mermaid
flowchart TD
    A[Worker picks up job] --> B[Attempt to process]
    B --> C{Processing succeeded?}
    C -- Yes --> D[Write output to Postgres]
    D --> E[Confirm token debit in ledger]
    E --> F[Emit WebSocket completed]

    C -- No --> G[Catch error]
    G --> H[INSERT INTO job_retry_history]
    H --> I{attempt_number < max_attempts?}

    I -- Yes --> J[Calculate backoff delay]
    J --> K[Re-enqueue job with delay in BullMQ]
    K --> L[UPDATE bullmq_jobs SET status: retrying, next_retry_at]
    L --> M[Emit WebSocket retrying]

    I -- No --> N[Job exhausted all retries]
    N --> O[INSERT INTO dead_letter_jobs]
    O --> P[UPDATE bullmq_jobs SET status: dead]
    P --> Q[INSERT token_ledger debit_released - restore credits to user]
    Q --> R[UPDATE study_set_batches SET failed_jobs += 1]
    R --> S{All jobs in batch failed?}
    S -- Yes --> T[UPDATE study_set_batches SET status: failed]
    S -- No --> U[Batch partially completed]
    T --> V[Emit WebSocket batch_failed]
    U --> V
```

## Group 4: Token Ledger Flows

### 4.1 — Token ledger update after AI job completion

```mermaid
sequenceDiagram
    participant Worker
    participant ModelRouter as Model Router
    participant Postgres
    participant Ledger as Token Ledger

    Worker->>ModelRouter: Request AI generation
    ModelRouter-->>Worker: output + raw_tokens + model_name + provider

    Worker->>Postgres: SELECT token_multiplier FROM model_config WHERE model_name = ? AND provider = ?
    Postgres-->>Worker: multiplier value

    Worker->>Worker: computed_credits = raw_tokens * multiplier rounded up

    Worker->>Ledger: INSERT token_ledger {event_type: debit_confirmed, amount: -computed_credits, raw_provider_tokens: raw_tokens, multiplier, job_id, idempotency_key}

    Worker->>Ledger: SELECT amount FROM token_ledger WHERE job_id = ? AND event_type = debit_pending
    Ledger-->>Worker: reserved_credits (negative amount)

    Worker->>Worker: delta = reserved_credits - computed_credits

    alt actual cost less than reservation
        Worker->>Ledger: INSERT token_ledger {event_type: refund, amount: +delta, job_id, note: reservation_excess}
    end

    alt actual cost exceeded reservation
        Worker->>Ledger: INSERT token_ledger {event_type: debit_confirmed, amount: -delta, job_id, note: over_estimate_correction}
    end

    Worker->>Postgres: UPDATE users SET token_balance = token_balance - computed_credits
```

### 4.2 — Monthly credit grant (cron job)

```mermaid
sequenceDiagram
    participant Cron as Cron Scheduler
    participant Backend
    participant Postgres
    participant Ledger as Token Ledger

    Cron->>Backend: Trigger monthly_credit_grant job at billing cycle start

    Backend->>Postgres: SELECT subscriptions WHERE status = active AND current_period_start = TODAY
    Postgres-->>Backend: List of qualifying subscriptions

    loop For each active subscription
        Backend->>Postgres: SELECT * FROM monthly_credit_grants WHERE user_id = ? AND billing_period_start = ?

        alt Grant already processed (idempotency check)
            Backend->>Backend: Skip — already granted this period
        end

        Backend->>Postgres: SELECT monthly_credit_allotment FROM plans WHERE id = subscription.plan_id

        Backend->>Postgres: SELECT rollover_policy FROM plans WHERE id = ?

        alt Credits do not rollover
            Backend->>Postgres: SELECT SUM(amount) FROM token_ledger WHERE user_id = ?
            Backend->>Ledger: INSERT token_ledger {event_type: credit_grant, amount: -current_balance, note: period_reset}
        end

        Backend->>Ledger: INSERT token_ledger {event_type: credit_grant, amount: +monthly_allotment, note: monthly_renewal}
        Backend->>Postgres: UPDATE users SET token_balance = new_balance
        Backend->>Postgres: INSERT INTO monthly_credit_grants {user_id, subscription_id, credits_granted, processed: true}
        Backend->>Postgres: INSERT INTO audit_logs {action: monthly_credit_grant, target: user_id}
    end
```

### 5.2 — Stripe webhook processing (all event types)

```mermaid
flowchart TD
    A[POST /payments/webhook arrives] --> B[Verify Stripe-Signature header with webhook secret]
    B --> C{Signature valid?}
    C -- No --> D[Return 400 - reject]
    C -- Yes --> E[Parse event type]
    E --> F{Already processed?}
    F -- Yes --> G[Return 200 idempotent skip]
    F -- No --> H[INSERT stripe_webhook_events status: processing]

    H --> I{Event type?}

    I -- checkout.session.completed --> J[Create subscription + grant credits]
    I -- invoice.payment_succeeded --> K[Renew subscription + grant monthly credits]
    I -- invoice.payment_failed --> L[UPDATE subscription status: past_due]
    I -- customer.subscription.deleted --> M[UPDATE subscription status: cancelled]
    I -- customer.subscription.updated --> N[UPDATE plan tier if changed]

    J --> O[UPDATE stripe_webhook_events status: processed]
    K --> O
    L --> O
    M --> O
    N --> O

    O --> P[INSERT audit_logs]
    P --> Q[Return 200 OK to Stripe]

    B --> R{Processing throws error?}
    R -- Yes --> S[UPDATE stripe_webhook_events retry_count += 1]
    S --> T{retry_count < 5?}
    T -- Yes --> U[Return 500 - Stripe will retry]
    T -- No --> V[UPDATE status: failed - alert admin_notifications]
```

## Group 6: AI Model Router Flows

### 6.1 — Model selection and provider routing

```mermaid
flowchart TD
    A[Worker calls Model Router with task_type] --> B[SELECT from model_config WHERE task_type = ? AND is_active = true]
    B --> C{rollout_percentage < 100?}
    C -- Yes --> D[Hash user_id to integer mod 100]
    D --> E{hash mod 100 < rollout_percentage?}
    E -- No --> F[Use previous stable model from config]
    E -- Yes --> G[Use new rollout model]
    C -- No --> G

    G --> H[Build request payload for provider]
    F --> H

    H --> I{Provider?}
    I -- OpenAI --> J[POST api.openai.com/v1/chat/completions]
    I -- Anthropic --> K[POST api.anthropic.com/v1/messages]
    I -- Gemini --> L[POST generativelanguage.googleapis.com]

    J --> M{Response OK?}
    K --> M
    L --> M

    M -- Yes --> N[Extract output + raw_tokens]
    N --> O[Compute credits = raw_tokens * multiplier]
    O --> P[Return output + computed_credits + model_meta to Worker]

    M -- No --> Q{Error type?}
    Q -- Rate limit 429 --> R[Wait backoff_ms then retry once]
    R --> M
    Q -- Provider down 5xx --> S[Load fallback_model from model_config]
    S --> T[INSERT into ai_request_logs status: fallback_used]
    T --> H
    Q -- Bad request 4xx --> U[Return error to Worker - do not retry]
```

### 6.2 — Model rolling upgrade procedure

```mermaid
sequenceDiagram
    participant Admin
    participant Backend
    participant Postgres
    participant ModelRouter as Model Router

    Admin->>Backend: POST /admin/model-config {task_type, new_model, provider, multiplier, rollout_percentage: 10}
    Backend->>Postgres: INSERT INTO model_config {task_type, model_name, is_active: true, rollout_percentage: 10}
    Backend->>Postgres: INSERT INTO audit_logs {action: model_rollout_started}
    Backend-->>Admin: 201 Created - 10% of traffic will use new model

    Note over ModelRouter: 10% of users get new model based on uid hash

    Admin->>Backend: GET /admin/model-performance {task_type, model_name}
    Backend->>Postgres: SELECT avg latency, error_rate, avg_credits FROM ai_request_logs WHERE model_name = ?
    Backend-->>Admin: Performance metrics comparison

    alt New model performing well
        Admin->>Backend: PATCH /admin/model-config {id, rollout_percentage: 100}
        Backend->>Postgres: UPDATE model_config SET rollout_percentage = 100
        Backend->>Postgres: UPDATE old model_config SET is_active = false
```
