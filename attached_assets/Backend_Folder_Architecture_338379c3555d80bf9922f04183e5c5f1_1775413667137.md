# Project Structure & Folder Layout

---

## Technology decisions reflected in this structure

- Runtime: Node.js with TypeScript
- Framework: Fastify (chosen over Express for native async support, schema validation, and better performance)
- ORM: Drizzle ORM (chosen over Prisma for raw SQL proximity, better TypeScript inference, and no query engine binary)
- Queue: BullMQ with Redis
- Validation: Zod (for request/response schema validation, shared with Drizzle types)
- Testing: Vitest
- Package manager: pnpm
- Monorepo: No — single backend service, workers are separate processes within the same repo

## Key files explained in detail

---

### `src/app.ts`

This is the Fastify application factory. It does not start the server — it builds and returns the configured Fastify instance. This separation allows tests to import the app without binding to a port.

```
Responsibilities:
- Creates Fastify instance with logger, schema validator (ajv), and request ID generation
- Registers @fastify/cors with allowed origins from config
- Registers @fastify/multipart for PDF file uploads
- Registers @fastify/helmet for security headers
- Registers the WebSocket plugin
- Registers the global error handler middleware
- Registers the request logger middleware
- Registers all route modules in order
- Returns the configured app instance
```

---

### `src/server.ts`

The actual entry point. Imports the app factory, binds to port, handles graceful shutdown signals.

```
Responsibilities:
- Calls app factory to get configured Fastify instance
- Initialises database connection and verifies connectivity
- Initialises Redis connection and verifies connectivity
- Binds to PORT from config (default 3000)
- Listens for SIGTERM and SIGINT — on signal, drains in-flight requests, closes DB pool, closes Redis, exits cleanly
```

---

### `src/workers/index.ts`

This file is the entry point for worker processes. A worker process is started with an environment variable `WORKER_TYPE=notes` (or any task type). This single entry point reads that variable and starts the appropriate worker class. In production, each worker type runs as its own process or container.

```
Responsibilities:
- Reads process.env.WORKER_TYPE
- Imports the corresponding worker class
- Instantiates it and calls .start()
- Handles uncaught exceptions — logs and exits with code 1
```

---

### `src/workers/base.worker.ts`

All worker classes extend this. It handles all the cross-cutting concerns so individual workers only implement the `process()` method.

```
Responsibilities:
- Instantiates BullMQ Worker connected to the correct queue
- On job received: updates bullmq_jobs status to processing in Postgres
- Calls this.process(job) — implemented by subclass
- On success: calls this.onSuccess(job, result)
  - Stores output to study_set_outputs
  - Calls ledger service to finalize token debit
  - Emits WebSocket event via Redis pub/sub
  - Updates bullmq_jobs to completed
  - Updates study_set_batches completed_jobs count
- On failure: calls this.onFailure(job, error)
  - Logs error to job_retry_history
  - If retries remaining: re-enqueues with backoff, updates bullmq_jobs to retrying
  - If all retries exhausted: writes dead_letter_jobs, releases token debit_pending, marks bullmq_jobs as dead
- Heartbeat: writes worker_health row to Postgres every 30 seconds
```

### `src/websocket/ws.emitter.ts`

Workers never hold WebSocket connections — they only emit events by publishing to a Redis pub/sub channel. The Fastify server subscribes to those channels and forwards to connected WebSocket clients. This architecture means workers and the API server are fully decoupled.

```
Responsibilities:
- Exports a single emit(channel, event) function
- Publishes JSON-serialised event to Redis channel key: ws:batch:{batch_id}
- Called by base.worker.ts after every job state change
```

---

### `src/websocket/ws.server.ts`

Runs inside the Fastify process. Subscribes to Redis channels and forwards to connected WebSocket clients.

```
Responsibilities:
- On WebSocket connection: authenticates the ephemeral token
- Extracts batch_id from the connection URL
- Subscribes to Redis channel ws:batch:{batch_id}
- On Redis message: finds all WebSocket clients for that channel, sends the event
- On client disconnect: unsubscribes from Redis channel if no other clients remain
- Handles ping/pong keepalive
```

### `src/services/model-router/model-router.service.ts`

The single abstraction between the entire application and all AI providers.

```
Responsibilities:
- Accepts ModelRequest { task_type, prompt, user_id, job_id, max_tokens }
- Queries model_config table for active model for this task_type
- If rollout_percentage < 100, hashes user_id to determine which variant the user gets
- Calls the correct provider client (openai, anthropic, or gemini)
- If provider returns 429: waits backoff_ms, retries once
- If provider returns 5xx: loads fallback_model from config, logs fallback event, calls fallback provider
- Returns ModelResponse { output, raw_tokens, model_name, provider, latency_ms }
- Does NOT write to the ledger — the caller (worker) does that
```

---

### `src/services/rag/rag.service.ts`

Handles all vector search operations. Called by workers before building their prompt.

```
Responsibilities:
- Accepts RAGRequest { document_id, task_type, top_k, user_id, job_id }
- Builds a task-specific query string (e.g. for flashcards: "key terms, definitions, concepts worth memorising")
- Calls model router to embed the query string using the embedding model
- Runs pgvector cosine similarity search on document_chunks for the given document_id
- Orders results by similarity, selects top_k chunks
- Re-orders selected chunks by original chunk_index to preserve reading flow
- Trims assembled context if total token count exceeds model context limit
- Writes to rag_contexts and vector_search_logs tables
- Returns assembled_context string
```

---

### `src/outputs/outputs.stream.ts`

Handles the SSE streaming endpoint for notes. This is the only route in the system where the AI response is streamed directly to the client rather than stored first.

```
Responsibilities:
- Validates output_id and verifies ownership
- Checks output status — if already completed, fetches stored markdown and streams it as deltas immediately (replays from DB)
- If output is still being processed, opens a streaming request to the AI provider via model router stream mode
- Sets response headers: Content-Type: text/event-stream, Cache-Control: no-cache, Connection: keep-alive
- Sends metadata event first
- Pipes AI response stream chunk by chunk as delta events
- On stream complete: saves full assembled markdown to study_set_outputs, writes to ledger, sends done event
- On stream error: sends error event, marks output as failed, releases pending ledger entry
```

## `Dockerfile.worker` — worker processes

```docker
FROM node:20-alpine AS base
WORKDIR /app
RUN npm install -g pnpm

FROM base AS deps
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile --prod

FROM base AS build
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN pnpm build

FROM base AS prod
COPY --from=build /app/dist ./dist
COPY --from=deps /app/node_modules ./node_modules
ARG WORKER_TYPE
ENV WORKER_TYPE=${WORKER_TYPE}
CMD ["node", "dist/workers/index.js"]
```

In production, each worker type is deployed as its own container with `WORKER_TYPE` set as a build arg or environment variable — so `notes`, `podcast`, `embedding`, and `grader` each scale independently based on queue depth.
